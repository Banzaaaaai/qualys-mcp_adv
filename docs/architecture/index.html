Architecture

Internal architecture of the Qualys MCP Server — caching, concurrency, request deduplication, and tool routing.

System Overview

The Qualys MCP Server is a single-process FastMCP application that exposes 35 tool functions to AI assistants. Each tool call flows through caching, optional concurrent execution, and response assembly before returning structured data.

Cache Architecture

The server uses a tiered in-memory cache to balance freshness against API call volume. Each tier reflects the volatility of its data: operational data changes frequently and gets short TTLs, while reference data and authentication tokens use longer windows.

Cache TTL Key Strategy
Bearer token 3.5 hours Single global token
KB entries 1 hour Per-QID
VMDR detections 5 minutes Per severity_days_qds_min (limit excluded)
QDS scores 5 minutes Per-QID
WAS findings 10 minutes Per limit_severity_days_app_name
Scanner list 5 minutes Single global list
ETM results 1 hour Single global (unfiltered only)

The three tiers map to data volatility:

  • L1 — Hot (5-minute TTL): Volatile operational data such as detections, QDS scores, and scanner lists. These change frequently and must stay fresh.
  • L2 — Warm (1-hour TTL): Slow-changing reference data like KB entries and ETM report results. These rarely change within an hour.
  • L3 — Cold (3.5-hour TTL): Authentication tokens. The bearer token is refreshed only when it expires.

Request Deduplication

When multiple tool calls fire concurrently (e.g., inside get_morning_report), they may request the same underlying data. The _get_or_fetch pattern uses per-key locks to ensure only one thread fetches while others wait for the result:

def _get_or_fetch(cache_key: str, fetch_fn, ttl: int = 300):
    """Get from cache or fetch with request deduplication.

    If two threads request the same uncached key simultaneously,
    only one will fetch -- the other waits and gets the result.
    """
    # Fast path -- cache hit, no lock needed
    hit, data = _cache_get(cache_key)
    if hit:
        return data

    # Slow path -- serialize with per-key lock
    if cache_key not in FETCH_LOCKS:
        FETCH_LOCKS[cache_key] = Lock()

    with FETCH_LOCKS[cache_key]:
        # Double-check after acquiring lock
        hit, data = _cache_get(cache_key)
        if hit:
            return data

        # Actually fetch
        data = fetch_fn()
        if data is not None:
            _cache_set(cache_key, data, ttl)
        return data

This double-check locking pattern prevents duplicate API calls when the same data is requested by parallel threads within _run_concurrent().

Concurrency Model

The server uses ThreadPoolExecutor(max_workers=8) via the _run_concurrent() helper to parallelize independent API calls within a single tool invocation. Not all tools benefit from concurrency — some make only one API call.

Sequential Tools

Tools that make a single API call or require serial data dependencies run sequentially: investigate_cve, get_qid_details, get_new_vulns, get_webapp_vulns, get_expiring_certs, get_edr_events, get_fim_events, and others.

Tool Routing

There is no explicit router or coordinator tool. Instead, the LLM selects tools based on enhanced docstrings that include routing hints. Each tool docstring follows a pattern that tells the LLM when to use it and when not to:

"""[Risk Management] Weekly top risks...
Use when asked: 'what should we fix this week', 'top vulnerabilities',
'prioritize', 'what are our biggest risks', 'what to patch first'
NOT for: single CVE lookups (use investigate_cve),
morning updates (use get_morning_report)
"""

This approach avoids the latency of a meta-routing layer while giving the LLM enough context to pick the right tool on the first attempt.

Aggregator Tools

Three tools act as aggregators, combining data from multiple sources in a single call:

  • get_asset(detail="full") — Combines CSAM metadata, ETM confirmed findings, and VMDR host detections for a single asset via parallel calls (~5-8s cold, ~2s warm).
  • get_morning_report(quick=True) — Fast all-CSAM environment snapshot (<3s): total assets, OS breakdown, cloud split, EOL counts, criticality distribution.
  • get_risk_by_tag(tag) — Risk distribution for a tagged asset group: count, risk tiers, top risky assets, EOL count via parallel CSAM queries (~3s).

Performance Budget

The target is a 10-second total response time for any single tool call. The budget breaks down as follows:

Phase Budget What Happens
Token check 0ms Memory lookup
Parallel API calls 3–8s Actual Qualys API calls
Data processing <0.5s XML/JSON parsing
Response assembly <0.1s Dict construction

Key bottleneck mitigations:

  • VMDR classic API (~2min) — Use ETM instead for detection queries. Cached ETM reports return in <1s.
  • ETM report creation (~1-5min) — Always cache the last completed report with a 1-hour TTL.
  • Cloud evaluation loops (~10s) — Parallelize provider fetches across AWS, Azure, and GCP.
  • WAS findings — 10-minute cache on get_webapp_vulns.
