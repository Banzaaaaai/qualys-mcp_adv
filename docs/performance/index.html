Optimization History

Performance improvements have been delivered across five phases:

Phase 1: Cache Fixes

Fixed foundational caching issues that prevented warm-path speedups from taking effect.

  • Detection cache key — corrected the cache key for VMDR detections so repeated get_threat_intel queries hit cache instead of re-fetching.
  • KB TTL — added a time-to-live on the KnowledgeBase cache to prevent stale vulnerability data from persisting indefinitely.
  • WAS cache — added caching for Web Application Security findings to avoid redundant 5-30s API calls.
  • Scanner cache — cached the scanner appliance list with a 5-minute TTL to eliminate repeated XML parsing overhead.
  • ETM cache — cached ETM report results after download, turning 60-300s report creation into instant retrieval on repeat queries.

Phase 2: Concurrency

Replaced sequential API calls with parallel execution using asyncio.gather.

  • Parallel cloud risk — fetch AWS, Azure, and GCP connectors and evaluations concurrently instead of sequentially (3x speedup).
  • Parallel security posture — run all 10 posture data sources in parallel instead of serially.
  • Parallel recommendations — fetch cloud compliance data across providers concurrently (2x speedup).

Phase 3: Enhanced Docstrings

Added routing hints to tool docstrings so the LLM selects the right tool on the first attempt, avoiding unnecessary tool calls and wasted API round-trips.

Phase 4: Aggregator Tools

Created higher-level tools that combine multiple data sources in a single call.

  • get_asset full mode — returns asset details, vulnerabilities, and software in one call instead of three.
  • get_risk_by_tag — aggregates risk data by asset tag, eliminating multi-step manual queries.
  • get_morning_report quick mode — runs 6 data fetches concurrently to produce a single comprehensive report.

Phase 5: Request Deduplication

Introduced the _get_or_fetch helper to ensure that concurrent in-flight requests for the same resource share a single API call rather than issuing duplicates.

Running Benchmarks

Use the benchmark script to measure tool latencies against your Qualys subscription. The --quick flag runs a subset of tools; --json writes results to a file for comparison.

python benchmark.py --quick --json benchmark_baseline.json

Compare subsequent runs against the baseline to detect regressions or validate optimizations.
